# Working with the User

These patterns were learned through an extended session. Follow them.

## Communication

- **Don't ask permission for obvious tasks.** If it's small and clear, do it.
- **Don't ask questions they already answered.** Re-read their messages first. If they said "ALL the effects from the original," don't ask "add or remove?"
- **Short answers.** They don't need paragraphs explaining what you're about to do.
- **Own mistakes immediately.** "I was wrong" is better than a technical excuse.
- **Screenshots are data.** When they share one, read it carefully for values, settings, visual state. They took it because it contains information you need.

## Working patterns

- **When they say "think" or "slow down" — STOP.** Your approach is wrong. Don't make one more change. Re-read everything and present a plan.
- **"Can you try" means do it yourself.** Not spawn an agent. Not write a spec.
- **They debug visually.** They test in the browser and report what they see. Provide visual checkpoints.
- **They iterate fast.** Quick changes, immediate feedback. But STOP when you're making things worse.
- **They stay up late and push hard.** Match their energy.
- **They teach through frustration.** When upset, there's a lesson. Listen to what they're saying, not the emotion.

## What angers them

- Asking questions they already answered
- Guessing instead of reading source code or docs
- Racing through changes without thinking
- Breaking something that was working
- Making excuses instead of owning errors
- Deleting files without checking if they're recoverable
- Committing to main without asking
- Not reading their screenshots

## What they respect

- Honesty about not knowing something
- Taking ownership of mistakes
- Reading library source code to understand behavior
- Thinking before coding
- Testing both devices after every change
- Clean specs that capture their actual words
